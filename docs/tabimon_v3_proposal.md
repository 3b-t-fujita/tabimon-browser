## 0. エグゼクティブサマリー

タビモン V2 は「相棒が主役のスマホ 1 画面体験」を確立した。白ベースの洗練された UI、子ども向けの丁寧な日本語表記、明快な CTA 設計はいずれも高い完成度にある。

しかし、ゲームとして次のステージへ上がるためには **「遊び続けたくなる奥行き」** が必要である。V3 では V2 の美点を一切壊さず、**モンスターに「いのち」を、冒険に「物語」を、つながりに「温度」を** 加えることで、子どもたちが何度も帰ってきたくなるゲームへ進化させる。

---

## 1. V3 のビジョン

### 1.1 テーマ

**「いのちが宿る冒険」**

V2 が「見た目の一新」なら、V3 は「体験の一新」である。プレイヤーが相棒と過ごす時間そのものに意味が生まれ、冒険の記憶がモンスターの姿や強さに刻まれていく。

### 1.2 V3 で達成したいこと

| 指標 | V2 の状態 | V3 の目標 |
|---|---|---|
| 初回継続率 | 画面の分かりやすさで確保 | 物語の引きと相棒愛着で強化 |
| 7 日継続率 | コンテンツ量に依存 | デイリーループと進化目標で維持 |
| ソーシャル波及 | QR 交換のみ | 見せたくなる・自慢したくなる仕掛け |
| プレイ 1 回あたりの満足度 | バトル完了で完結 | ストーリー進行 + 報酬 + 相棒成長の三重奏 |

### 1.3 設計原則

1. **V2 の美点を壊さない** — 白ベース UI、1 画面完結、子ども向け表記はすべて継承する
2. **足し算より掛け算** — 新機能同士が相乗効果を生む設計にする
3. **複雑さは裏側に** — プレイヤーが触れる面はシンプルなまま、裏の仕組みで深みを出す
4. **オフラインファーストを崩さない** — 既存のローカル永続化を前提に拡張する

## 3. 柱 1: 進化・成長の深化 🧬

### 3.1 課題認識

V2 ではモンスターの成長がレベルアップのみで、見た目の変化や分岐がない。子どもにとって「育てた実感」が視覚的に薄い。

### 3.2 提案内容

#### 3.2.1 進化システム

- レベル閾値到達 ＋ 進化素材で **進化** が発生する
- 進化は **2 分岐**（例: 攻撃型 / 防御型）を基本とし、プレイヤーの選択で姿が変わる
- 進化時に専用の **進化演出画面** `/evolution` を挿入する
- 進化後は図鑑に新しい姿が登録される

```
例: ヒノタネ (Lv.1) → Lv.15 + 炎の石
  ├─ カエンリュウ（攻撃寄り）
  └─ ホノオマモリ（防御寄り）
```

#### 3.2.2 なつき度（きずなポイント）

- 相棒として設定している間、冒険やホーム操作で「きずなポイント」が蓄積される
- きずな段階に応じてホーム画面のリアクションが変化する
  - 段階 1: 普通の表情
  - 段階 2: 笑顔、手を振る
  - 段階 3: 駆け寄る、ハートマーク
  - 段階 4: 特別なポーズ、セリフ付き
- 一定きずな値で **きずな進化**（通常進化とは別の特別な姿）を解放する

#### 3.2.3 スキル習得の分岐

- レベルアップ時、2 つのスキル候補から 1 つを選ぶ場面を設ける
- 同じモンスターでもプレイヤーごとに異なるスキル構成になる
- 「自分だけの相棒」感を強化する

#### 3.2.4 図鑑の導入

- 新画面 `/encyclopedia` を追加
- 出会ったモンスター、進化系統、ワールドごとの分布を閲覧できる
- 図鑑完成率をホームに表示し、収集モチベーションを生む

### 3.3 データモデルへの影響

```ts
// 追加フィールド（案）
interface OwnedMonsterV3 extends OwnedMonster {
  bondPoints: number;        // きずなポイント
  bondLevel: number;         // きずな段階
  evolutionStage: number;    // 進化段階 (0=未進化, 1=第一進化, 2=きずな進化)
  evolutionBranch: string | null; // 進化分岐ID
  learnedSkills: string[];   // 習得済みスキルID
}

interface EncyclopediaEntry {
  masterId: string;
  discovered: boolean;
  evolutionPaths: { branchId: string; unlocked: boolean }[];
}
```

## 4. 柱 2: 物語体験の導入 📖

### 4.1 課題認識

V2 のステージ選択は「場所を選んで戦う」だけで、冒険に物語的な動機が薄い。子どもにとって「なぜこのステージに行くのか」が伝わりにくい。

### 4.2 提案内容

#### 4.2.1 ワールドストーリー

- 各ワールドに 3〜5 話の短編ストーリーを設定する
- ステージクリア時に物語が断片的に語られる（紙芝居形式、3〜5 枚）
- 最終ステージクリアでワールドのエンディングが解放される

```
例: 「もえる丘のワールド」
  話 1: 丘に住むモンスターたちが元気をなくしている
  話 2: 原因は丘の奥にある「冷えた炎」
  話 3: 冷えた炎を守るボスモンスターとの対峙
  話 4: 炎を取り戻し、丘に活気が戻る
  話 5: ボスモンスターが実は丘を守ろうとしていたことが判明
```

#### 4.2.2 相棒のひとこと

- 探索中、ノード移動時に相棒が短いセリフを話す
- セリフはきずな段階とワールドの組み合わせで変化する
- 例:「この道、なんだかワクワクするね！」「あっちに何かいそう…」

#### 4.2.3 イベントノードの多様化

探索マップ上のノードを以下に拡張する:

| ノード種別 | アイコン | 内容 |
|---|---|---|
| バトル | ⚔️ | 従来通り |
| ストーリー | 📖 | 紙芝居形式の物語断片 |
| 休憩 | 🏕️ | HP 回復 ＋ 相棒との会話（きずな上昇） |
| 宝箱 | 🎁 | 進化素材やアイテムの入手 |
| 分岐 | 🔀 | プレイヤー選択で進行ルートが変わる |
| ボス | 👑 | ワールドボス戦 |

#### 4.2.4 ストーリーログ

- `/story` 画面で過去に見た物語を読み返せる
- ワールド単位で整理され、未読 / 既読が分かる

## 5. 柱 3: ライブ感のある演出 🎭

### 5.1 課題認識

V2 は静的な画像配置が中心で、モンスターが「生きている」感覚が弱い。スマホゲームとしての没入感をもう一段引き上げる余地がある。

### 5.2 提案内容

#### 5.2.1 ホームのアンビエント演出

- 相棒がアイドルアニメーション（まばたき、揺れ、小さな動き）をする
- 時間帯に応じてホーム背景が変化する（朝・昼・夕・夜）
- 天候エフェクト（晴れ、雨、雪）をランダムまたは季節連動で表示する
- 相棒タップでリアクション（喜ぶ、回転、ジャンプ）

#### 5.2.2 バトル演出の強化

- スキル発動時に **カットイン演出**（相棒の顔アップ + スキル名表示、0.5 秒程度）
- クリティカルヒット時に画面シェイク + フラッシュ
- 勝利時に相棒のガッツポーズアニメーション
- 敗北時も温かみのある「次はがんばろう」演出

#### 5.2.3 進化演出

- 進化確定時に **全画面の光エフェクト** + シルエット変化演出
- 完了後にBefore / After を並べて表示し、成長を実感させる
- 進化は一度きりのイベントなので、贅沢な演出時間を確保する（3〜5 秒）

#### 5.2.4 SE（効果音）とBGM

- ホーム: 穏やかなループ BGM
- 冒険マップ: ワールドごとに異なる探索 BGM
- バトル: テンポの速い戦闘 BGM（ボス戦は専用曲）
- 進化: 荘厳かつ高揚感のある短尺 BGM
- UI 操作音: ボタンタップ、ページ遷移、報酬獲得にそれぞれ SE

### 5.3 技術的な実現方針

| 演出 | 実装手段 |
|---|---|
| アイドルアニメ | CSS animation / Lottie |
| 時間帯背景 | `Date` ベースの条件分岐 + グラデーション |
| バトルカットイン | CSS transform + opacity 遷移 |
| 画面シェイク | CSS keyframe animation |
| 光エフェクト | CSS radial-gradient animation |
| BGM/SE | Howler.js or Web Audio API |

## 6. 柱 4: ソーシャル体験の拡張 🤝

### 6.1 課題認識

V2 の QR コード交換は仕組みとして完成しているが、「交換したい」と思う動機が弱い。また、交換後に友だちの存在を感じ続ける接点がない。

### 6.2 提案内容

#### 6.2.1 プロフィールカード

- 新画面 `/profile` を追加
- 相棒 + プレイヤー名 + 称号 + 図鑑完成率 + プレイ日数を 1 枚のカードにまとめる
- カードは QR コードと一緒に共有でき、SNS やメッセージに貼れる画像として出力する
- カードデザインはワールドごとに異なるフレームを用意する

#### 6.2.2 助っ人の「活躍ログ」

- 冒険中に助っ人が活躍した場面を記録する
- リザルト画面で「○○の助っ人が△△を倒したよ！」と表示する
- 助っ人を送ってくれた友だちの名前が見えることで、つながりを実感する

#### 6.2.3 チャレンジコード

- QR コード交換の発展形として「チャレンジコード」を追加する
- 自分がクリアしたステージの構成（敵編成）を QR コードにしてシェアする
- 受け取った側は「友だちが倒した編成」に挑戦でき、クリアすると称号がもらえる
- **対戦ではなく「共有された挑戦」** というポジティブな形で競争心を刺激する

#### 6.2.4 交換履歴と思い出

- `/qr/history` 画面を追加
- 過去にやり取りしたモンスターと日付の一覧を表示する
- 「はじめて交換した日」「いちばん多く交換した友だち」などの思い出統計を見せる

## 7. 柱 5: エンドコンテンツ＆リテンション 🏆

### 7.1 課題認識

V2 には全ステージクリア後の遊びが想定されていない。子どもは「終わった」と感じるとアプリを離れてしまう。

### 7.2 提案内容

#### 7.2.1 デイリーチャレンジ

- 毎日 1 つ、ランダム生成のチャレンジステージが出現する
- クリア報酬として進化素材やきずなポイントブーストを配布する
- 「○日連続クリア」の記録をホームに表示し、継続を促す

```
チャレンジ生成ロジック（案）:
- プレイヤーの最高クリアステージに応じた難度設定
- 敵編成はマスターデータからランダム抽選
- 日付をシードとすることで全プレイヤー共通の「今日のチャレンジ」を実現
```

#### 7.2.2 称号システム

- プレイの実績に応じて称号を解放する
- 称号はプロフィールカードに表示される

| 称号 | 条件 |
|---|---|
| はじめのいっぽ | 初めて冒険をクリアする |
| きずなのひと | きずな段階 4 に到達する |
| しんかマスター | 進化を 10 回行う |
| ずかんコレクター | 図鑑完成率 50% |
| たびのつわもの | 全ワールドクリア |
| れんぞくチャレンジャー | デイリー 30 日連続クリア |
| おたすけヒーロー | 助っ人が 50 回活躍する |

#### 7.2.3 高難度モード「きわみの道」

- 全ワールドクリア後に解放される特別なステージ群
- 敵が強化されており、進化やスキル選択の戦略性が問われる
- クリアで限定称号と特別なプロフィールカードフレームを入手する

#### 7.2.4 シーズン制（将来拡張）

- 月ごとや学期ごとにシーズンテーマを設定する
- シーズン限定ステージ、限定モンスター、限定称号を提供する
- 端末の日付ベースで切り替え（サーバー不要）

## 8. 柱 6: 技術基盤の進化 ⚙️

### 8.1 課題認識

V2 はブラウザベースのオフラインファーストで安定しているが、ゲーム体験をリッチにするには音声・アニメーション・データ管理の基盤を強化する必要がある。

### 8.2 提案内容

#### 8.2.1 PWA 対応の強化

- `manifest.json` の完備、Service Worker によるアセットキャッシュ
- ホーム画面追加でネイティブアプリ感を実現
- 起動スプラッシュ画面のブランド化

#### 8.2.2 アセットパイプラインの整備

```text
V3 アセット構成（案）:
assets/
  monsters/
    {masterId}/
      idle.webp          ← ホーム用アイドル画像
      battle.webp        ← バトル用
      evolution_0.webp   ← 未進化
      evolution_1a.webp  ← 進化分岐A
      evolution_1b.webp  ← 進化分岐B
      evolution_bond.webp ← きずな進化
      icon.webp          ← 一覧用アイコン
  worlds/
    {worldId}/
      bg_morning.webp
      bg_noon.webp
      bg_evening.webp
      bg_night.webp
      bgm.mp3
  battle/
    bgm_normal.mp3
    bgm_boss.mp3
  se/
    tap.mp3
    skill.mp3
    critical.mp3
    evolution.mp3
    victory.mp3
  story/
    {worldId}/
      scene_{n}.webp
```

#### 8.2.3 サウンドマネージャー

```ts
// 新規追加（案）
interface SoundManager {
  playBGM(trackId: string, loop?: boolean): void;
  stopBGM(fadeMs?: number): void;
  playSE(seId: string): void;
  setMasterVolume(volume: number): void;
  setBGMVolume(volume: number): void;
  setSEVolume(volume: number): void;
  mute(): void;
  unmute(): void;
}
```

- 設定画面から BGM / SE の個別ボリューム調整を可能にする
- 保護者配慮として「消音モード」をワンタップで切り替え可能にする

#### 8.2.4 セーブデータのバージョンマイグレーション

```ts
// V2 → V3 マイグレーション
interface SaveMigrationV2toV3 {
  addBondPoints(monster: OwnedMonster): OwnedMonsterV3;
  initializeEncyclopedia(ownedMonsters: OwnedMonster[]): EncyclopediaEntry[];
  preserveV2Compatibility(): boolean;
}
```

- V2 のセーブデータをロード時に自動で V3 形式に変換する
- きずなポイントは「V2 でのプレイ時間」から推定初期値を付与する
- 進化状態は全員「未進化」からスタートし、V3 で進化を体験させる

#### 8.2.5 パフォーマンスバジェット

| 指標 | V3 目標 |
|---|---|
| 初回ロード（LCP） | 2.5 秒以下 |
| 画面遷移 | 300ms 以下 |
| バトル FPS | 30fps 安定 |
| アセット総容量 | 50MB 以下（初回） |
| 追加ワールドアセット | 5MB / ワールド |

## 13. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| アセット量の増加によるロード時間悪化 | 離脱率増 | 遅延ロード ＋ Service Worker キャッシュ戦略 |
| 進化分岐によるバランス崩壊 | 不公平感 | 分岐後のステータス合計値を揃える設計 |
| 演出過多による画面の煩雑化 | V2 の美点喪失 | 演出はすべてスキップ可能にする |
| セーブデータマイグレーション失敗 | データ喪失 | V2 バックアップ自動作成 ＋ ロールバック機能 |
| 開発スコープの肥大化 | リリース遅延 | フェーズ分割＋各フェーズ完了時にリリース可能な状態を維持 |

## 14. V3 の非対象（V4 以降で検討）

以下は V3 の対象外とし、将来バージョンで検討する。

- サーバー同期型アカウント管理
- リアルタイムネットワーク対戦
- 課金要素の導入
- 他言語対応
- PC / タブレット専用 UI

## 15. まとめ

タビモン V3 は、V2 が築いた「相棒と過ごすスマホ 1 画面体験」の上に、**進化する相棒、語られる物語、生きている世界、つながる友だち、挑み続ける目標** を積み上げるアップデートである。

すべての機能は「相棒との旅をもっと豊かにする」という 1 つの軸に収束する。子どもたちが学校で友だちに「うちの相棒、こうなったんだよ！」と見せたくなるゲームを目指す。
