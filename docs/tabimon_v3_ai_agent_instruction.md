# タビモン V3 初回リリース AIエージェント向け実装指示書

作成日: 2026-03-28  
対象: `tabimon-browser`  
前提: `tabimon_v3_decision_sheet.md` を最優先の正本として扱うこと

---

## 1. この指示書の目的

この文書は、AIエージェントがタビモン V3 初回リリースを実装するための**着手用実行指示書**である。

目的は以下の3点。

1. スコープを広げすぎないこと
2. データ基盤から先に安全に実装すること
3. 実装完了後、依頼者がすぐに動作確認に入れる状態を作ること

---

## 2. 最重要ルール

## 2-1. スコープ固定

今回の実装対象は、以下のみに限定すること。

- EXP表示強化
- きずなシステム
- スキル熟練システム
- ステージ選択の `ものがたり / そだてる` 分離
- ホーム画面改修
- 仲間詳細画面改修
- ステージ選択画面改修
- リザルト画面改修
- 軽量進化演出
- V2 → V3 データマイグレーション

以下は**実装しないこと**。

- 分岐進化の本実装
- きずな進化
- 新しい設定画面
- 図鑑の本格改修
- BGM / SE 強化
- PWA強化
- `/evolution` 新規ページ
- 大量アニメーション演出

---

## 2-2. 実装方針

- 演出より先にデータ反映を作る
- UIは既存のV2デザインシステムに揃える
- 既存導線を壊さず、必要最小限の改修で対応する
- 既存セーブデータを壊さない
- 新規ルートは原則増やさない
- 1機能ずつビルド可能な単位で進める

---

## 3. 実装完了時に満たすべき状態

以下が揃っていることをゴールとする。

### 必須

- backend が不要な構成なら frontend build 成功
- TypeScript エラーなし
- 主要画面の手動確認が可能
- V2セーブデータからV3へ移行可能
- ホームで相棒・EXP・きずなが見える
- 仲間詳細で EXP / きずな / 熟練が見える
- ステージ選択で `ものがたり / そだてる` が見える
- リザルトで EXP / きずな / 熟練 / 進化通知 が見える
- 既存の冒険導線が壊れていない

### できれば

- テスト追加
- README更新
- 確認観点メモ
- 触るべきURL一覧
- サンプルデータや検証条件メモ

## Phase 2. データモデル導入

以下を追加・更新すること。

### 4-2-1. OwnedMonsterV3

`OwnedMonster` に対して以下の追加を行う。

```ts
currentExp: number;
bondPoints: number;
bondRank: 0 | 1 | 2 | 3 | 4;
skillProficiency: Record<string, {
  useCount: number;
  stage: 0 | 1 | 2 | 3;
}>;
evolutionBranchId?: string | null;
bondMilestoneRead?: number[];
```

#### ルール

- `bondRank` は `bondPoints` から再計算可能であること
- `skillProficiency` は `useCount` を主データとすること
- 既存参照箇所を壊さないように型変更を波及対応すること

### 4-2-2. StageMasterV3

```ts
stageType: 'STORY' | 'FARM';
farmCategory?: 'EXP' | 'BOND' | 'SKILL';
difficultyTier?: 'EARLY' | 'LATE';
recommendedLevel?: number;
estimatedMinutes: number;
firstClearBonusExp?: number;
```

### 4-2-3. DailyRecord

```ts
interface DailyRecord {
  date: string;
  homeTapCount: number;
}
```

### Phase 2 の成果物

- 型定義更新
- 永続化スキーマ更新
- コンパイル通過

## Phase 3. マイグレーション実装

V2セーブデータからV3へ移行する処理を実装すること。

### ルール

- 既存モンスターの `bondPoints = 0`
- `bondRank = 0`
- 全スキル熟練 `useCount = 0`, `stage = 0`
- 既に進化済みの個体はそのまま維持
- 既存図鑑進行は維持
- 失敗時に旧データへ戻せる設計を優先

### 必須対応

- マイグレーションバージョン管理
- 旧データ退避
- 一度移行済みなら再実行しない
- 不足項目は安全な初期値で補完

### Phase 3 の成果物

- migration 実装
- 旧セーブからの移行確認
- 新規セーブ生成確認

---

## Phase 4. ドメインルール実装

以下の計算・反映処理を実装すること。

### 4-4-1. きずなルール

#### 段階閾値

- 0: 0
- 1: 50
- 2: 150
- 3: 400
- 4: 1000

#### 付与ルール

- ホームタップ: 1日3回まで `+1`
- 冒険完了時: 相棒に所定値加算
- 失敗時も加算してよいが、成功時より小さくする

※ 冒険完了時の付与値は、既存バランスに合わせて実装時に定数化すること。ハードコード散在は禁止。

### 4-4-2. 熟練ルール

- スキル使用時に `useCount +1`
- `useCount` に応じて `stage` を更新

#### 閾値

- 0: 0
- 1: 5
- 2: 15
- 3: 30

### 4-4-3. 進化ルール

- 既存の Lv15 進化を維持
- 分岐進化は実装しない
- 初回は単一路線のみ

### Phase 4 の成果物

- 純関数またはポリシーとして整理された計算ロジック
- UI層にロジックを散らさない実装

---

## Phase 5. リザルト反映の改修

**最優先画面** としてリザルトを先に作ること。

### 必須表示

1. EXPセクション
2. きずなセクション（相棒のみ）
3. スキル熟練セクション
4. ドロップセクション
5. 進化通知または軽量進化演出
6. CTA

### 必須要件

- 今回の冒険で「何が進んだか」がひと目で分かること
- レベルアップ通知が出ること
- きずな段階アップ通知が出ること
- 熟練上昇が分かること
- 進化条件を満たした場合に通知・演出が出ること

### 遷移

- 候補あり → `/adventure/candidate`
- 候補なし → `/home`

### Phase 5 の成果物

- リザルトUI更新
- 反映ロジック更新
- 導線確認

---

## Phase 6. 主要画面改修

## 6-1. ホーム画面

### 必須要件

- 相棒が主役に見える
- 相棒カード全体から詳細へ遷移できる
- EXP要約表示
- きずな要約表示
- 軽いタップリアクション
- `ぼうけんへ しゅっぱつ` が主CTA

### タップ仕様

- 1日3回まで `+1`
- 4回目以降は見た目反応のみ

---

## 6-2. 仲間詳細画面

### 必須要件

- EXPの詳細表示
- きずなの詳細表示
- スキルごとの熟練表示
- 既存の相棒設定・手放し等を壊さない

---

## 6-3. ステージ選択画面

### 必須要件

- `/adventure/stages` 内でタブ切替
- `ものがたり` と `そだてる` を分離表示
- `そだてる` 側で `EXP / BOND / SKILL` と `前半 / 後半` が分かること
- 推奨Lv・主報酬・目安時間を表示

### ルール

- 新規ルート追加は禁止
- 2タブは同一画面内で実装すること

---

## Phase 7. 軽量演出実装

## 7-1. きずな演出

初回は軽量版のみ。

### 許容範囲

- 表情差分
- 小さなハートや吹き出し
- テキスト通知

### 禁止

- 重い連続アニメーション
- 大量の新規アセット前提実装

## 7-2. 進化演出

初回は短く気持ちよく見えることを優先。

### 許容範囲

- フェード
- フラッシュ
- Before / After の簡易表示
- テキスト通知

### 禁止

- 新規 `/evolution` ページ
- 長尺演出
- 分岐選択UI

## 5. 実装時の注意点

### 5-1. やってはいけないこと

- 提案書ベースでスコープを広げること
- 初回から分岐進化を入れること
- データ移行前に画面だけ先行で作ること
- ロジックをコンポーネント内に散らすこと
- 既存導線を大きく壊すこと

### 5-2. 優先すべきこと

- セーブ互換
- 型安全
- 主要導線維持
- MVP成立

## 7. エージェントへの最終メッセージ

今回の実装で重要なのは、V3を豪華にすることではなく、**初回V3をきちんと成立させること**です。

判断に迷った場合は、以下の順で優先してください。

1. セーブ互換を守る
2. データ反映を正しくする
3. リザルトを成立させる
4. ホーム・詳細・ステージ選択を整える
5. 演出は軽量でまとめる

広げすぎず、壊さず、でも体験は確実に良くしてください。
