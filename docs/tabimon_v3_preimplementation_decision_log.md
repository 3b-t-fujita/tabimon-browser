# タビモン V3 実装前に決めるべき論点一覧

> 注意: この資料は実装前の検討ログです。`そだてる` の旧カテゴリ案を含みます。現行判断は [tabimon_v3_decision_sheet.md](./tabimon_v3_decision_sheet.md) を正本としてください。

作成日: 2026-03-28  
対象: タビモン V3 UI/演出改修 実装着手前の意思決定整理

## この資料の目的

本資料は、AIエージェントに実装指示を出す前に、
**未確定のままだと実装中に手戻り・迷い・設計ぶれが起きやすい論点** を整理するためのものです。

各論点について、以下の4項目で整理しています。

- **論点**: 決めるべきテーマ
- **なぜ決める必要があるか**: 実装・設計・体験面での理由
- **推奨判断**: 現時点での現実的なおすすめ
- **まだ保留でもよいか**: 実装前に必須か、後回し可能か

---

## 1. スコープ・実装範囲

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| 初回V3に含める機能範囲 | ここが曖昧だと、AIエージェントが安全側で広く実装しようとしてスコープ肥大化する | 初回V3は **EXP / きずな / 熟練 / ステージ分離 / 進化演出軽量版 / リザルト強化** に限定する | **不可** |
| 初回V3で分岐進化を入れるか | UI、データ構造、進化演出、マスタ設計が一気に重くなる | **初回は単一路線優先**。分岐進化は受け皿だけ用意し、本実装は後続 | **不可** |
| ホームの本格リアクション演出をどこまで入れるか | 表情差分、エフェクト、音、アニメで負荷が大きく変わる | **初回は軽量版**（表情差分 + 吹き出し + 小エフェクト）まで | **不可** |
| 図鑑画面改修を初回V3に含めるか | 既存画面改修まで広げると、表示・導線・整合テストが増える | **初回は含めない**。既存図鑑は維持し、進化時の図鑑追加通知だけ対応 | 可 |
| 設定画面 / BGM / SE / PWA強化を今回含めるか | 実装コストが高く、ゲーム体験の中核ではない | **今回の実装対象外**。別パッチ扱い | 可 |
| 進化確認専用画面 `/evolution` を新設するか | URL設計、画面遷移、戻り先設計に影響する | **初回は新設しない**。リザルト内フローまたはモーダルで完結 | **不可** |

## 2. データモデル・保存設計

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| `OwnedMonsterV3` の最終項目 | UI/演出より先に保存構造が固まっていないと後戻りコストが高い | `currentExp`, `bondPoints`, `bondRank`, `skillProficiencyUses`, `skillProficiencyStage` を持つ | **不可** |
| `bondRank` を保存するか都度計算するか | UI全体で何度も参照されるため、方針次第で実装量とバグ率が変わる | **保存してよい**。ただし基準値から再計算可能な設計にして整合チェックを入れる | **不可** |
| 熟練を uses で持つか stage だけ持つか | 将来の調整幅が変わる | **uses を主データ**、stage は導出値または保存補助にする | **不可** |
| 分岐進化用の履歴や選択結果を初回から持つか | 今後の拡張性に関わるが、今入れると設計が重くなる | 初回は **持たない**。必要なら `evolutionBranchId?: string` の受け皿だけ許容 | 可 |
| きずな演出の既読・直近セリフ履歴を永続化するか | 演出の重複制御に関わる | 初回は **最小限**。`lastShownBondDialogueIds` は任意、段階アップ通知の既読だけ持つ | 可 |
| `StageMasterV3.stageType` の定義 | ステージ選択UIの2タブと直結する | `STORY` / `FARM` で固定する | **不可** |
| `farmCategory` の定義 | `そだてる` タブの見せ方に直結する | `EXP` / `EVOLUTION_MATERIAL` / `RECRUIT` で固定する | **不可** |
| 目安時間をメタデータで持つか | UIに出すなら出典が必要 | `estimatedMinutes` をマスタに持つ | **不可** |
| `DailyRecord` の範囲 | 日次制御の責務が曖昧だと、あとで日替わり処理が崩れる | 初回は **`homeTapCount` のみ必須**。必要なら将来拡張 | **不可** |
| 日付判定基準 | 端末依存の仕様は、日付切替やずれに影響する | 初回は **端末ローカル日付基準** で割り切る | **不可** |

## 3. V2 → V3 マイグレーション

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| 既存モンスターのきずな初期値をどうするか | 既存プレイヤー体験と公平感に影響する | **全員0開始** または **簡易補填値** のどちらかを明確に決める。おすすめは全員0開始で説明を簡潔にする | **不可** |
| 既に進化済み個体の扱い | 巻き戻すとユーザー違和感が大きい | **進化状態は維持**。V3演出は今後進化する個体で体験させる | **不可** |
| 熟練の初期値 | 新システムの導入感に関わる | **全スキル0開始** でよい | **不可** |
| 図鑑進行を引き継ぐか | プレイヤー資産保護に関わる | **引き継ぐ** | **不可** |
| マイグレーション失敗時の扱い | セーブ破損時の回復策が必要 | 旧データ退避 + バージョンフラグ付き移行 + 失敗時ロールバック | **不可** |

## 4. バランス値・可変値

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| どの数値を固定値として実装するか | 実装後の調整容易性が大きく変わる | 数値群は `BALANCE_V3` 等の定数管理に集約する | **不可** |
| きずな段階値 `50 / 150 / 400 / 1000` を固定するか | UI、通知、進行感に影響する | **固定で実装**。調整は後続パッチで見直し | **不可** |
| 熟練段階値 `15 / 40 / 80` を固定するか | 仲間詳細・リザルト・バランス全体に影響する | **固定で実装** | **不可** |
| `そだてる` ステージEXP値を確定するか | UI文言やおすすめ判定に影響する | 初回は現仕様値で **固定** | **不可** |
| 初回クリアボーナスを固定するか | ストーリー進行テンポに影響する | 初回は現仕様値で **固定** | **不可** |
| 進化後 `きずな +30` を固定するか | リザルト・進化演出・きずな段階通知の整合性に関わる | **固定で実装** | **不可** |
| アンダーレベル / オーバーレベル補正の調整余地 | 周回効率と育成導線に影響する | ロジックは実装するが、補正値は定数化して調整可能にする | **不可** |

## 5. UI / 演出の軽量版・本格版の境界

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| ホーム演出の到達点 | ここが曖昧だと、演出だけ豪華で他が未完成になりやすい | 初回は **軽量版のみ**。ジャンプや回転、音はやらない | **不可** |
| きずな演出のセリフ量 | 多すぎると管理が破綻する | 初回は **共通文 + 場面別最小セット** で十分 | **不可** |
| 表情差分の必要枚数 | アセット工数に直結する | 初回は **通常 / 笑顔 / うれしい / しょんぼり / やる気** の5種目安 | **不可** |
| 進化演出の長さ | 長すぎると周回テンポを壊す | **短尺 + スキップ可** を固定する | **不可** |
| 分岐進化候補の見せ方 | 分岐を入れる場合のアセット量が変わる | 初回で分岐をやらないなら保留。やるなら **2択 + 3比較項目まで** に固定 | 可 |
| 音・BGM前提で設計するか | 無音環境でも成立する必要がある | **音なしでも成立するUI/演出** を前提にする | **不可** |

## 6. 画面遷移・URL設計

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| `/adventure/stages` のタブ状態をURLに持つか | 戻る操作・共有・保守性に影響する | 初回は **UI stateのみ** でよい。必要ならクエリ追加は後で対応 | 可 |
| 進化確認を独立URLにするか | 画面数と遷移複雑度が増える | **独立URLにしない**。リザルト導線内で完結 | **不可** |
| `/settings` を今回出すか | スコープ拡大に直結する | **出さない** | 可 |
| `/encyclopedia` を今回触るか | 導線整理と図鑑改修の有無に関わる | **今回は触らない** | 可 |
| ステージ選択から編成確認への導線 | UI実装と遷移の境界を決める必要がある | 既存どおり **ステージ選択 → 冒険確認 → 冒険開始** を維持 | **不可** |

## 7. アセット要件

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| 進化演出用の新規画像が必要か | 実装が画像待ちで止まる可能性がある | 初回は **既存立ち絵 + 軽い演出** で成立させる | **不可** |
| きずな用表情差分の必要枚数 | ここが未確定だと演出実装が進まない | 5種程度で固定 | **不可** |
| `ものがたり / そだてる` 用の専用背景やバッジ素材が必要か | UI差別化に影響する | 初回は **既存UI部品 + 色 / ラベル差分** で対応 | **不可** |
| 熟練の見た目変化をどう表現するか | アセット有無で実装方式が変わる | 初回は **テキスト + 星表示** のみ | **不可** |
| BGM/SEアセットを今回準備するか | 制作工数が大きい | **不要** | 可 |

## 8. 文言・表記ルール

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| ひらがな優先の範囲 | UIのトーン統一に必要 | 子ども向け主要UIは **ひらがな中心**、技術的概念や固有名は必要最低限で漢字可 | **不可** |
| `あいぼう / 相棒` などの表記統一 | 文言ブレは品質を下げる | UI表示は **あいぼう** に統一 | **不可** |
| `しんか / 進化` の表記統一 | 場面ごとに混ざると読後感がぶれる | UI表示は **しんか**、内部資料や設計書は `進化` で可 | **不可** |
| CTA文言のトーン | 押しやすさに影響する | **行動がすぐ分かる短文** で統一（例: `ぼうけんへ しゅっぱつ`） | **不可** |
| 通知文の長さ | 吹き出しや通知カードのUI崩れに影響する | 原則 **1行、長くても2行まで** | **不可** |

## 9. 完成判定・プレイテスト観点

| 論点 | なぜ決める必要があるか | 推奨判断 | まだ保留でもよいか |
|---|---|---|---|
| 何をもって「完成」とするか | AIエージェントは実装できても、合格基準がないと判断できない | 画面表示だけでなく **体験品質基準** を先に置く | **不可** |
| ホームの合格基準 | V3の顔になる画面だから | **3秒以内に主CTAが分かる / 相棒が主役に見える** を基準にする | **不可** |
| 仲間詳細の合格基準 | 情報画面が説明書化しやすい | **EXP / きずな / 熟練の違いが説明なしで伝わる** | **不可** |
| ステージ選択の合格基準 | タブ分離が形だけになるリスクがある | **ものがたり / そだてる の違いが一目で分かる** | **不可** |
| リザルトの合格基準 | V3報酬体験の受け皿だから | **何が伸びたかが嬉しく分かる** | **不可** |
| 進化演出の合格基準 | 長すぎ・弱すぎの両リスクがある | **短いのに特別感がある / スキップしても破綻しない** | **不可** |

---

## 10. いま決めるべき優先順位まとめ

### 実装前に必ず決める
1. 初回V3スコープ
2. データモデル最終仕様
3. V2→V3マイグレーション方針
4. 進化分岐対応有無
5. 画面遷移 / URL設計
6. バランス値の固定範囲
7. アセット必要数
8. 文言 / 表記統一ルール
9. 完成判定基準

### あとで決めてもよい
1. 図鑑改修
2. 設定画面
3. BGM / SE / PWA
4. 分岐進化の全面展開
5. モンスター個別セリフの大量追加
6. 共有機能や称号などの次点要素

---

## 11. 総評

現状は「仕様が弱い」のではなく、**仕様がかなり固まってきたからこそ、実装前に最終判断すべき論点が見えている状態**です。

特に重要なのは、
- 何を今回入れるか
- どこまで軽量版で割り切るか
- データと遷移をどう固定するか

この3つです。

ここを先に決めてからAIエージェントに渡すと、実装中の迷いがかなり減り、
V3の主導権を仕様側で握ったまま進めやすくなります。
