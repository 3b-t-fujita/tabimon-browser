# Tabimon V3 実行計画

## 目的
- V3 初回リリースに向けて、主要導線の安定性と確認性を高める
- `setup` / `home` / `stages` を中心に、ブラウザ実行時の不安定さを減らす
- Playwright を使った主要導線の自動確認を整備する

## ステップ 1. 開発環境差分の解消
- `next dev` と `next start` で挙動差が出る箇所を切り分ける
- 少なくとも `setup` と `home` は両環境で同じ操作感になるようにする
- 画面遷移、submit、初回表示が片方だけ壊れる状態を解消する

## ステップ 2. `/home` 読込まわりの安定化
- 初回起動、既存セーブ、移行直後の3パターンで `/home` の読み込み経路を確認する
- 「読み込み中のまま止まる」状態を再現・解消する
- セーブ無し時の `setup` 誘導、セーブ有り時のホーム表示を明確にする

## ステップ 3. V3 主要フローの E2E 拡張
- Playwright の対象を以下まで広げる
  - `setup -> home`
  - `home -> adventure/stages`
  - `adventure/stages` の `ものがたり / そだてる` 切替
  - `そだてる` 6 ステージ構成の表示確認
  - `preview/ui`
  - `monster detail`
  - `result` の主要表示
- 失敗時に原因が追いやすいセレクタと期待値を採用する

## ステップ 4. V3 データ反映の通し検証
- `EXP`, `きずな`, `熟練`, `初回クリアEXP`, `進化通知` が表示だけでなく実データに反映されるか確認する
- `EXP / BOND / SKILL` の前半 / 後半差が実データへ反映されるか確認する
- 主要フローで「何が増えたか」が分かることを重視する
- 必要に応じてユースケーステストを追加・修正する

## ステップ 5. マイグレーション確認
- V2 セーブから V3 セーブへの移行を複数パターンで確認する
- 不足項目補完、既存進行維持、再実行安全性を検証する
- 既存セーブを壊さないことを最優先にする

## ステップ 6. リリース前整理
- 確認 URL、確認手順、既知の注意点、Playwright 実行方法を文書化する
- 手動確認と自動確認の両方でリリース判断できる状態にする
- README または docs の参照導線を整える

## 推奨の分担
- 担当 A: `setup` / `home` 安定化、環境差分の解消
- 担当 B: Playwright 拡張、E2E 実行と失敗分析
- 担当 C: データ反映確認、マイグレーション確認、統合と最終報告

## 完了条件
- `setup` からホームまでの通常開始が安定して通る
- `/home` の読み込み停止が再現しにくい状態になる
- Playwright の主要スモークが通る
- Vitest と build が通る
- 確認手順と既知事項が文書化される
